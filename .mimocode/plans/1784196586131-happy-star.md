# Portfolio Chatbot — Rate Limit & Token Optimization Plan

## Current State Analysis

After reading `ChatBot.jsx` (982 lines), here are the findings:

### What's Currently Happening

| Setting | Current Value | Impact |
|---|---|---|
| `max_tokens` | **512** (line 172) | Moderate — allows ~1300 tokens per chat |
| System prompt | **~500-600 tokens** (lines 7-108) | LARGE — this is the #1 token consumer |
| Message history | **Unbounded** — sends ALL prior messages every call (line 475) | CRITICAL — every new chat sends the entire conversation |
| Model | `llama-3.3-70b-versatile` | Fine, but heavier on tokens than smaller models |
| Streaming | ✅ Enabled | Good |
| 429 handling | ✅ Implemented with cooldown banner | Good |

### Why You're Hitting Rate Limits

1. **Unbounded message history** — line 475: `const history = newMessages.map(...)` sends the ENTIRE chat history every time. After 10 messages, each API call already costs ~3000+ tokens just in history.

2. **Large system prompt** — your system prompt is ~550 tokens. Every single API call includes this.

3. **`max_tokens: 512`** — not terrible, but you don't need it for a portfolio chatbot.

### Token Math (Current State)

```
Per chat:
  System prompt:     ~550 tokens
  User message:      ~20 tokens
  Assistant response: up to 512 tokens (max)
  Overhead:          ~100 tokens (formatting)
  ─────────────────────────
  Total per chat:    ~680-1180 tokens

With 10-message history:
  History alone:     ~1500 tokens
  + system + response: ~2230 tokens total

Daily limit: 100,000 tokens
  At ~800 tokens/chat (short history): ~125 chats/day
  At ~2200 tokens/chat (long history): ~45 chats/day
```

## Recommended Changes

### Change 1: Reduce `max_tokens` from 512 → 400
**File:** `ChatBot.jsx` line 172

**Why 400 (not 300):**
- 300 is aggressive and may cut off answers about projects with multiple bullet points
- 400 gives comfortable room for 3-4 sentence answers with a project detail or two
- Still saves 22% vs current 512
- For a portfolio chatbot, this is the sweet spot

**Impact:** ~22% fewer tokens per response.

### Change 2: Limit message history to last 10 messages
**File:** `ChatBot.jsx` line 475

**Change from:**
```js
const history = newMessages.map(m => ({ role: m.role, content: m.content }));
```

**To:**
```js
const recentMessages = newMessages.slice(-10);
const history = recentMessages.map(m => ({ role: m.role, content: m.content }));
```

**Why:** This is the BIGGEST win. After a 20-message conversation, the current code sends all 20 messages + system prompt every call. With the cap, you send at most 10 + system prompt. This prevents the "token death spiral" where each message gets progressively more expensive.

**Impact:** Caps history at ~1500 tokens regardless of conversation length.

### Change 3 (Optional): Shorten system prompt
**File:** `ChatBot.jsx` lines 7-108

This is a larger change. The system prompt is well-written but ~550 tokens. You could compress it to ~350 tokens by:
- Removing redundant phrasing
- Condensing project descriptions (keep tech + one-liner, drop full descriptions)
- Removing the "IMPORTANT: Never use markdown symbols" instruction (move to a shorter note)

**Impact:** ~35% fewer system prompt tokens, but requires rewriting the prompt.

### Change 4 (Optional): Conditional system prompt
**File:** `ChatBot.jsx` `buildSystemPrompt` function (line 152)

Currently adds a `CURRENT PAGE CONTEXT` section. This is smart — keep it. But consider: when the section is `hero`, don't append anything (line 153-154 already does this). ✅ Already optimized.
